# M001/S04 — Research

**Date:** 2026-03-16
**Depth:** Targeted

## Summary

S04 delivers five enhancements to the blog reading experience: TOC with scroll-spy (R008), related posts (R010), share buttons (R011), code block copy button (R009 partial), and tag archive pages (R006). All consume existing infrastructure from S02 — the blog content collection, BlogCard component, BlogPost layout, and reading time utility.

The riskiest piece is the TOC scroll-spy. R008 requires a sticky sidebar on desktop with active-heading highlighting, only for posts >1000 words. This forces a layout restructure of `BlogPost.astro` from single-column centered to a two-column layout (content + sidebar) on desktop when TOC is present. The scroll-spy itself is a standard IntersectionObserver pattern. Everything else — share buttons, related posts, copy button, tag archives — is straightforward Astro component work using established patterns.

One data gap: no existing posts exceed 1000 words (the longest is `markdown-style-guide.md` at ~671 body words). The executor needs to extend or create a post that crosses the 1000-word threshold to prove the TOC renders conditionally.

## Recommendation

Build in four tasks ordered by dependency and risk:

1. **Tag archives + tag pill links** — Creates `/blog/tag/[tag]/[...page].astro` and updates tag pills in BlogCard/BlogPost to link there. Establishes the tag navigation system. No client JS.
2. **Related posts + share buttons** — Two static components integrated into BlogPost.astro. Both are pure Astro, no client JS. Related posts queries the collection by shared tags; share buttons construct URLs.
3. **Copy button** — Client JS island that adds copy-to-clipboard on `.astro-code` blocks. Small, independent.
4. **TOC with scroll-spy** — The most complex piece. Extracts headings via `render()`, restructures post layout for sidebar, implements IntersectionObserver scroll-spy. Needs a >1000-word sample post.

Task 4 goes last because it modifies the BlogPost layout that tasks 2 and 3 also touch. Building it last means the layout is stable when the TOC sidebar restructure happens.

### Files to Create

- **`src/pages/blog/tag/[tag]/[...page].astro`** — Tag archive page. Uses `getStaticPaths` to enumerate unique tags from all posts, then `paginate()` filtered posts per tag. Reuses `BlogCard`. Pattern matches `[...page].astro` pagination exactly.
- **`src/components/TableOfContents.astro`** — Accepts `headings` array (from Astro `render()`). Renders nested nav of h2/h3 links. Client-side script uses IntersectionObserver to highlight active heading. Sticky positioning via Tailwind `sticky top-24`.
- **`src/components/RelatedPosts.astro`** — Accepts `currentSlug`, `currentTags`, and list of all posts. Filters by shared tags, excludes current, sorts by tag overlap count then date, takes top 3. Renders using a compact card format (not full BlogCard — too heavy for a "related" section).
- **`src/components/ShareButtons.astro`** — Accepts `url` and `title`. Renders share links for X (`https://twitter.com/intent/tweet?text=...&url=...`), LinkedIn (`https://www.linkedin.com/sharing/share-offsite/?url=...`), Dev.to (`https://dev.to/new?prefill=...`). Pure `<a>` tags with `target="_blank"` and `rel="noopener noreferrer"`.
- **`src/components/CopyButton.astro`** — Client JS island. On mount, finds all `.astro-code` elements, prepends a copy button to each. Click handler uses `navigator.clipboard.writeText()`. Shows brief "Copied!" feedback.

### Critical Data: Astro `render()` Headings

Astro's `render(post)` returns `{ Content, headings }` where `headings` is `Array<{ depth: number, slug: string, text: string }>`. The `slug` is auto-generated by Astro's markdown pipeline and corresponds to heading IDs in the rendered HTML. This is the data source for the TOC — no manual heading extraction needed.

Current `[slug].astro` does:
```ts
const { Content } = await render(post);
```
Must change to:
```ts
const { Content, headings } = await render(post);
```
Then pass `headings` to `BlogPost` layout as a prop.

## Constraints

- **Astro zero-JS default:** TOC scroll-spy and copy button require client JS. Use Astro island pattern — `<script>` tags in `.astro` components. No framework hydration needed (vanilla JS with IntersectionObserver).
- **Static Tailwind classes only:** Per K002 and S02 forward intelligence, dynamic class construction gets purged. All tag pill, TOC, and button styling must use complete static class strings.
- **`post.body` word count for TOC gating:** Use `getReadingTime()` from `reading-time.ts` — already computed in `[slug].astro`. A 1000-word post at 200 WPM = 5 min read. Pass readingTime to layout, gate TOC on `readingTime >= 5` (equivalent to ≥1000 words).
- **Heading scroll offset:** The sticky header takes space. Headings need `scroll-margin-top` in CSS so anchor clicks don't hide the heading behind the header.

## Common Pitfalls

- **Tag URL encoding** — Tags with spaces or special characters need proper URL encoding in routes. Use `encodeURIComponent()` for tag params in `getStaticPaths` and decode in the page. Current tags are all lowercase single-words so this isn't urgent, but the code should handle it.
- **IntersectionObserver root margin** — The scroll-spy must account for the sticky header height. Set `rootMargin: '-80px 0px -60% 0px'` (negative top for header, negative bottom to trigger earlier) so the active heading updates before it scrolls out of the top.
- **Two-column layout breakpoint** — TOC sidebar should only appear on `lg:` breakpoint and above. On mobile/tablet, the TOC (if shown at all) should collapse above the content or be hidden. Don't break mobile layout.
- **Empty related posts** — Posts with no tags or no tag overlap will have zero related posts. The component must handle this gracefully (render nothing, not an empty section).
- **Copy button positioning** — Code blocks use `overflow-x: auto` for horizontal scrolling. The copy button must be positioned absolutely relative to a wrapper, not inside the scrollable area, or it scrolls off-screen.
